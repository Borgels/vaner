# Vaner v0.6.0 install and usage findings (Linux)

## Environment

- Host: Linux 6.17.0-20-generic, bash.
- Prior install: `vaner 0.4.0` via `pipx`.
- Previous state quarantined to `~/.vaner.bak-20260421-133217`.
- Test sandbox: `/tmp/vaner-test-20260421-133253`.
- Installer tested: `https://vaner.ai/install.sh`.

## Install path findings

- Install script completed successfully and installed `vaner 0.6.0` to `/home/abo/.local/bin/vaner`.
- Script was idempotent on immediate rerun (fast verify/reinstall behavior).
- Installer attempted PyPI first, then fell back to GitHub source because `vaner[mcp]` was not resolvable from PyPI during test.
- Global MCP wiring happened automatically (`~/.cursor/mcp.json`) and also wrote skills to `~/.claude/skills/vaner/vaner-feedback/SKILL.md`.

## Init behavior findings

- `vaner init --path <repo>` wrote:
  - `<repo>/.vaner/config.toml`
  - `<repo>/.cursor/mcp.json`
  - `<repo>/.cursor/skills/vaner/vaner-feedback/SKILL.md`
  - `~/.claude/skills/vaner/vaner-feedback/SKILL.md`
- Existing files were preserved in-place:
  - Pre-existing `<repo>/AGENTS.md` unchanged.
  - Pre-existing `<repo>/.cursor/rules/something.mdc` unchanged.
- Re-running init repeatedly created additional `~/.cursor/mcp.json.vaner-backup-*` files (backup accumulation/noise).
- `--no-mcp` behaved as expected for MCP opt-out: no project `.cursor/` files were written.
- With global `~/.cursor/mcp.json` temporarily absent, init only configured project-local Cursor MCP files (did not recreate global file unexpectedly).

## CLI smoke findings

- Passing commands:
  - `vaner --version`
  - `vaner --help`
  - `vaner status`
  - `vaner doctor`
  - `vaner scenarios list`
  - `vaner scenarios show <id>`
  - `vaner up --path .` + `vaner down`
- Failing commands:
  - `vaner config show` crashes with `AttributeError: 'VanerConfig' object has no attribute 'intent'`.
  - `vaner config keys` crashes with same exception.
  - `vaner config set intent.lookback_turns 6` fails because key is unsupported and key discovery command is broken.
- Command surface mismatch vs earlier expectations:
  - No `vaner uninstall` command exists.
  - `vaner forget` exists and removes local state files, but is a coarse wipe rather than targeted cleanup.

## MCP surface and protocol findings

### Pre-reload context (legacy loaded surface)

- Cursor MCP descriptors exposed an older 5-tool shape:
  - `list_scenarios`
  - `get_scenario`
  - `expand_scenario`
  - `compare_scenarios`
  - `report_outcome`
- This does not match the expected v1 10-tool surface (`vaner.status`, `vaner.suggest`, `vaner.resolve`, `vaner.expand`, `vaner.search`, `vaner.explain`, `vaner.feedback`, `vaner.warm`, `vaner.inspect`, `vaner.debug.trace`).

### Post-reload direct MCP probe (critical)

- Direct stdio MCP client probe against `vaner mcp --path <sandbox>` failed at server startup:
  - `AttributeError: 'NoneType' object has no attribute 'tools_changed'`
  - crash path in `server.get_capabilities(notification_options=None, ...)`.
- Direct SSE MCP probe against `vaner mcp --transport sse` failed with same root cause:
  - SSE connection dropped (`incomplete chunked read`) after server-side capability initialization exception.
- Net effect: MCP server is not usable as a functioning MCP endpoint in this test environment after v0.6.0 install.

## System-prompt and takeover safety findings

- Init does not overwrite existing `AGENTS.md` or existing Cursor rule files in tested path.
- MCP-only mode is possible (`--no-mcp`) for users who do not want automatic client wiring.
- Global-vs-project behavior is mixed:
  - Installer/init may write global files (`~/.cursor/mcp.json`, Claude skill file).
  - Project-local wiring is also supported (`<repo>/.cursor/mcp.json`).
- There is no explicit `vaner uninstall`; rollback/undo is manual (restore backups and uninstall binary).

## Retention and DB growth findings

### Short burn test (sandbox)

- Before burst (50 `vaner query` calls + 20s daemon run):
  - `store.db`: 335872 bytes
  - `scenarios.db`: 40960 bytes
  - `metrics.db`: 53248 bytes
  - `telemetry.db`: 12288 bytes
- After burst:
  - `store.db`: 339968 bytes (+4096)
  - others unchanged in this short run

### Existing large repo state snapshot

- `/home/abo/repos/Vaner/.vaner/store.db` size: 213741568 bytes.
- Table counts in that store:
  - `artefacts`: 893
  - `signal_events`: 235866
  - `query_history`: 1
  - `prediction_cache`: 9
  - `hypotheses`: 19
- The dominant growth vector appears to be `signal_events`.

### Code-level retention behavior observed

- Purge hooks exist and are invoked in `Engine.precompute()`:
  - `purge_old_signal_events`
  - `purge_old_replay_entries`
  - `purge_old_query_history`
  - `purge_expired_prediction_cache`
  - `purge_stale_patterns`
- Retention window derives from `max_age_seconds` (minimum clamp to 3600 seconds).
- Practical gap: purge appears coupled to precompute/engine cycle, and there is no focused user-facing `prune/compact` CLI for targeted DB hygiene.

## Severity-ranked issues

### Broken

1. MCP server startup path breaks (stdio and SSE), preventing MCP tool use. (fixed in 0.6.1)
   - Repro: launch `vaner mcp` and initialize with MCP client.
   - Error: `notification_options=None` causing `tools_changed` access crash.
2. `vaner config show` and `vaner config keys` crash due config schema mismatch (`intent` attribute missing). (fixed in 0.6.1)

### Degraded

1. Public install path currently falls back to GitHub source because PyPI resolution for `vaner[mcp]` failed in this run. (hardened in 0.6.1 installer; release artifact verification still required)
2. Repeated init causes backup-file accumulation in `~/.cursor` (`mcp.json.vaner-backup-*` explosion). (fixed in 0.6.1)

### UX-smell

1. No dedicated uninstall/undo command; users must manually revert files. (fixed in 0.6.1 via `vaner uninstall`)
2. `vaner up` foreground behavior can leave long-running attached sessions unless user interrupts.
3. Help/flag expectations are not aligned with some previously documented/tested flags (for example `--clients`, `--minimal` not present).

### Nit

1. Installer output includes extension build instructions even when only CLI/MCP is needed.
2. Global skill writes to Claude occur by default, which may surprise users focused only on Cursor.

## Backlog items to pin

1. MCP server bootstrap fix for capability initialization (`notification_options` handling) and full end-to-end tool invocation tests in CI.
2. Config schema/CLI alignment fix (`config show|keys` crash).
3. Add explicit retention controls (`vaner prune` / `vaner compact`) plus clear docs for what is purged when.
4. Backup rotation policy for `~/.cursor/mcp.json.vaner-backup-*`.
5. Clarify install path strategy: PyPI-first vs GitHub fallback expectations.
6. Decide whether automatic global skill/config writes should be opt-in.
