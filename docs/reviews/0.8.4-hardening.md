## BLOCK findings

### BLOCK-1 — WS4 adoption-outcome log: rollback and rejection paths are wired but inert

**Files:** `engine.py:1835-1854`, `deep_run_maturation.py:861-885`, `config.py:567`, `engine.py:2543`.

`_reject_pending_adoptions_for_predictions` is implemented but has **zero call sites**. `rollback_kept_maturation` is also never invoked. The return value of `registry.apply_invalidation_signals(signals)` (outcomes dict including `"staled"` entries) is dropped on the floor at `engine.py:2543`. The config flag `adoption_rejection_on_stale` is read nowhere.

**Consequence:** the adoption-outcome log, in production, will only ever contain `pending` and `confirmed` rows — never `rejected` or `stale`. The `adoption_success_factor` scoring consumer (shipped gated behind `refinement.enabled`, which defaults False in 0.8.4) would therefore bias toward 1.5 for every ever-adopted prediction regardless of whether those predictions turned out wrong. That is the opposite of the WS4 design.

**Full fix landed in 0.8.4.** Both halves wired pre-tag after user review requested "fix things first."

1. **Stale half (`invalidation → "stale"`):** `_apply_cycle_invalidation` stashes `_last_staled_prediction_ids`; `precompute_cycle` drains it via `_resolve_pending_adoptions_for_predictions(..., outcome="stale", reason="invalidation_staled")`.

2. **Rejected half (`probation rollback → "rejected"`):** snapshot / rollback / reject pipeline:
   - `PredictionArtifacts` gained `pre_maturation_draft_answer` + `pre_maturation_evidence_score` fields.
   - `mature_one`'s kept branch now snapshots the pre-maturation draft + evidence score before overwriting. Snapshot lifetime is bounded by the probation window (≤ 3 cycles).
   - `_apply_cycle_invalidation` now captures `probation_snapshots` of every probationary prediction *before* applying signals. Any probationary prediction whose id appears in `apply_invalidation_signals`'s outcome map is rolled back via `rollback_kept_maturation(pred, rollback_to_draft=..., rollback_to_evidence_score=...)` using the snapshot, then added to `_last_rolled_back_prediction_ids`.
   - `rollback_kept_maturation` consumes the snapshot (clears both fields) after restore.
   - `precompute_cycle` drains `_last_rolled_back_prediction_ids` via `_resolve_pending_adoptions_for_predictions(..., outcome="rejected", reason="probation_rollback")`.
   - **Priority rule:** a prediction that is both in probation AND would have been staled is treated as `rejected`, not `stale`. The staled list excludes rolled-back ids so the same adoption outcome never flips twice.

3. **Tests:**
   - `test_mature_one_kept_path_persists_and_arms_probation` now asserts the snapshot is captured.
   - `test_rollback_restores_prior_draft_and_decrements_revision` now asserts the snapshot is consumed.
   - New engine-level `test_invalidation_during_probation_rolls_back_and_rejects_adoption` exercises the full path: probationary prediction + file_change signal → rollback + rejected adoption-outcome row.

With this in place the adoption-outcome log captures all three negative signals (`stale`, `rejected`, plus the existing `confirmed` positive) from day one. The refinement-flag scoring consumer (`adoption_success_factor`) can be activated cleanly in 0.8.5 with no additional wiring needed.

## HIGH findings (fix before tag unless noted)

### HIGH-1 — `rollback_kept_maturation` spurious permanent exclusion

**File:** `deep_run_maturation.py:861-885`.

Scenario: a prediction matures three times successfully (revision 0→1→2→3, `failed_revisits=0` each time). Reconciliation then contradicts three times → three rollbacks → revision walks back 3→2→1→0, but `failed_revisits` increments 0→1→2→3 on each rollback because the current code bumps it. With the balanced preset cap of 4, three more reconciliations would push `failed_revisits` to match the cap and **permanently exclude** the prediction from future maturation — even though it has zero persistent kept-and-wrong maturations.

**Fix:** rollback is semantically external ("the world changed under us"), not judge-discard ("the judge rejected this attempt"). Don't bump `failed_revisits` on rollback. Fixed.

### HIGH-2 — `rollback_kept_maturation` sets `last_matured_cycle` to the rollback cycle

**File:** `deep_run_maturation.py:885`.

Semantically backwards — the last-matured cycle just got undone. Future analytics ("when did this prediction last have a kept maturation") would read the rollback cycle as "the last maturation."

**Fix:** set `last_matured_cycle = None` after rollback. (We don't track history of prior matured cycles, so None is the honest value.) Fixed.

### HIGH-3 — `record_adoption` / `consume_pending_adoption_descriptors` race

**Files:** `prediction_registry.py:273-314`, `engine.py:1764`.

The MCP call site wraps `record_adoption` in `async with registry.lock`, but non-MCP call sites (`apply_invalidation_signals`, `apply_item_state_delta`) don't. `_flush_pending_adoption_outcomes` calls `consume_pending_adoption_descriptors` without holding the lock either. CPython's GIL makes `list.append` atomic in isolation, but `list(descriptors); descriptors.clear()` is two operations — appends between them get dropped.

**Fix:** use a dedicated `threading.Lock` for the descriptor queue. Hold it in both `record_adoption` (the append) and `consume_pending_adoption_descriptors` (the snapshot + clear). Fixed.

### HIGH-4 — `create_session` TOCTOU collapses clean error to IntegrityError

**File:** `store/deep_run.py:151-198`.

SELECT-then-INSERT in one connection without `BEGIN IMMEDIATE`. Two concurrent starts both see "no active session" and both reach INSERT; the UNIQUE partial index on `status='active'` catches the second, but as `IntegrityError` rather than the documented `DeepRunActiveSessionExistsError`.

**Fix:** catch `aiosqlite.IntegrityError` around the insert and re-raise as `DeepRunActiveSessionExistsError`. Also wrap in `BEGIN IMMEDIATE` so the second caller gets a lock wait instead of a failed commit. Fixed.

### HIGH-5 — `_consume_remote_budget` not atomic; crash-unblocks budget

**File:** `router/backends.py:137-160`.

`read_text` → parse → `write_text` is three syscalls with no lock or atomic-replace. Two concurrent callers can both read the same `used` value and both increment to `used+1`, overshooting `max_per_hour`. The `except Exception:` that resets state to 0 means a malformed state file unblocks the budget instead of fail-closed.

**Fix:** atomic write via `tempfile.NamedTemporaryFile` + `os.replace`. On parse failure, treat as cap-hit (return False) rather than reset. Fixed.

### HIGH-6 — `try_consume_cost` vs `reset_cost_gate` pointer-swap race

**File:** `intent/deep_run_gates.py:222-249`.

Thread A reads `state = _cost_state` (non-None, session X, its own lock). Thread B calls `reset_cost_gate(new_session)` rebinding `_cost_state` to a new object with a new lock. Thread A proceeds with `with state.lock:` on the old state — `spent_usd` is charged to the old session's counter, the new session's counter stays clean. First call after a session swap bypasses cost accounting by one charge.

**Fix:** read `_cost_state` under `_cost_state_lock` at the top of `try_consume_cost`, cost_gate_spent_usd, and remaining_cost_budget. Fixed.

### HIGH-7 — `refinement.require_idle_cpu` declared but never consulted

**File:** `models/config.py:563`, grep for "require_idle_cpu" across the repo → zero other references.

Dead config field. A user toggling it gets no behavior change.

**Fix:** remove the field. 0.8.4 ships with refinement off by default; no downstream consumers exist. 0.8.5 can re-add it with an actual use-site. Fixed.

### HIGH-8 — `adoption_pending_confirm_cycles` name says cycles; impl uses wall-clock

**File:** `engine.py:1812-1813`, `config.py:566`.

Docstring promises cycle-count semantics; code uses `time.time() - cfg.adoption_pending_confirm_cycles * 30.0` (hardcoded nominal cycle length). A user with `max_cycle_seconds=60` gets outcomes confirmed at twice the intended cycle count.

**Fix:** rename to `adoption_pending_confirm_seconds: float = 600.0`. Update docstrings and test. Name now matches behaviour. Fixed.

## MEDIUM findings

### Fixed in 0.8.4

- **MED-2 (Agent 1):** add `if candidate.prediction.is_terminal(): continue` check inside `_run_background_refinement_pass` loop, and re-check `prediction.is_terminal()` post-drafter-await inside `mature_one` before mutating run fields. Prevents corruption of a stale-transitioned prediction mid-maturation.
- **MED-5 (Agent 2):** add `await db.execute("PRAGMA busy_timeout=5000")` to all SQLite write paths in `store/deep_run.py` and `store/prediction_adoption_outcomes.py` (consistent with `create_session`'s existing pattern).
- **MED-6 (Agent 2):** add compound index `CREATE INDEX IF NOT EXISTS idx_adoption_outcomes_pending_adopted ON prediction_adoption_outcomes(outcome, adopted_at)` to speed `list_pending_outcomes`.
- **MED-7 (Agent 2):** chunk `update_pending_by_prediction_id` calls to ≤500 ids per statement, defensive against SQLite's `SQLITE_MAX_VARIABLE_NUMBER` default of 999.
- **MED-11 (Agent 3):** length-prefix the `prediction_label_hash` payload so `("foo|bar", "baz")` and `("foo", "bar|baz")` don't collide. Apply same to `prediction_id` for consistency.

- **MED-8 (Agent 2):** `replace_intent_artefact_items` DELETE + INSERT not transactional. Pre-existing 0.8.2 issue, not introduced by 0.8.4; backlog.
- **MED-9 (Agent 3):** `matured_rolled_back`, `matured_failed`, `promoted`, `explored` `DeepRunPassAction` values never assigned. Aspirational until 0.8.5 wires rollback + pass-log writes. CHANGELOG'd.
